Acrescente uma regra de tolerância a falhas para consultas externas.

Objetivo:
Se uma fonte externa falhar, o sistema não deve travar o processo inteiro de renomeação. A falha deve afetar apenas aquela fonte e aquele candidato, sem impedir que o arquivo seja analisado por outras fontes ou heurísticas.

Regra obrigatória:
Quando uma fonte externa falhar, o sistema deve:

1. registrar a falha da fonte;
2. indicar qual fonte falhou;
3. indicar, quando possível, o motivo da falha;
4. continuar consultando as demais fontes disponíveis;
5. reduzir a confiança do candidato afetado, se ele dependia daquela fonte;
6. não descartar automaticamente informações já obtidas de outras fontes;
7. não renomear automaticamente se a falha deixar a confiança abaixo do limite seguro;
8. marcar o arquivo para revisão manual quando a decisão ficar ambígua;
9. não interromper o processamento dos demais arquivos.

Tipos de falha a tratar:
- timeout;
- erro de conexão;
- resposta HTTP 429;
- resposta HTTP 500, 502, 503 ou 504;
- resposta inválida;
- JSON malformado;
- campo esperado ausente;
- fonte temporariamente indisponível;
- limite de requisições atingido.

Comportamento esperado:
- falha em Open Library não deve impedir consulta ao Google Books, Crossref ou outras fontes;
- falha em Google Books não deve impedir uso de metadados internos, nome do arquivo, ISBN, DOI ou outras fontes;
- falha em uma fonte não deve encerrar o script;
- falha em um arquivo não deve interromper o lote inteiro;
- o relatório final deve mostrar a falha da fonte e a decisão tomada.

Exemplo de relatório por arquivo:

source_failures:
  - source: openlibrary
    reason: timeout
    action: ignored_source_and_continued
confidence_adjustment:
  previous_score: 0.82
  final_score: 0.68
  reason: external_source_unavailable
action: review_required

Regras de implementação:

centralizar tratamento de falhas no cliente HTTP ou na camada de consulta externa;
não espalhar try/except genérico pelo código inteiro;
não usar except Exception: pass;
não esconder erro silenciosamente;
não transformar falha externa em metadado vazio sem registrar motivo;
usar erro estruturado ou objeto de resultado quando fizer sentido;
preservar o fluxo atual do script;
manter menor diff possível;
adicionar testes unitários com mocks para fontes externas falhando.

Testes obrigatórios:

uma fonte falha por timeout e as demais continuam;
uma fonte retorna HTTP 500 e o script continua;
uma fonte retorna JSON inválido e o erro é registrado;
todas as fontes externas falham e o arquivo é marcado como revisão manual ou skipped;
falha em um arquivo não interrompe processamento dos demais;
confiança é reduzida quando uma fonte relevante falha;
o sistema não renomeia automaticamente quando a confiança final fica baixa.

Restrições:

não alterar CLI;
não alterar padrão final de nomeação;
não adicionar dependência externa sem necessidade;
não reescrever o sistema de consulta externa;
não implementar crawler novo;
não mascarar erro;
não travar o renomeador inteiro por falha de fonte externa.

Antes de editar:

localize onde as fontes externas são chamadas;
localize onde o score/confiança é calculado;
localize onde o relatório por arquivo é montado;
proponha o patch mínimo;
indique os testes que serão criados.

Ao final:

liste arquivos alterados;
explique como cada tipo de falha é tratado;
liste os testes adicionados;
informe como rodar os testes;
destaque qualquer limitação restante.
